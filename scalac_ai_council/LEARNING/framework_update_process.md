# Framework Update Process
## Living Documentation

---

## Principle

"System prompt każdego agenta to 'living document'" - aktualizujemy go na bieżąco na podstawie:
- Nowych książek
- Nowych case'ów (CORE lub AI)
- Failed frameworks (analiza + modyfikacja)
- Quarterly review (czy 80/20 split still valid?)

---

## Update Triggers

### 1. New Book Read

**When:** After book study session  
**Who:** Agent who led the session  
**What:**
- Extract key concepts
- Create templates
- Update own system prompt
- Notify other agents about relevant updates

### 2. New Case Study

**When:** After project completion  
**Who:** Agent who worked on the project  
**What:**
- Add example to examples/ folder
- Update system prompt with new insights
- Document what worked/didn't work

### 3. Failed Framework

**When:** Framework didn't work in practice  
**Who:** Agent who used the framework  
**What:**
- Document failure in post-mortem
- Analyze why it failed
- Modify or remove framework
- Update system prompt

### 4. Quarterly Review

**When:** End of quarter  
**Who:** All agents + Head of Growth  
**What:**
- Review 80/20 split
- Update priorities
- Adjust system prompts

```

```

## Quality Checklist

Before updating system prompt:

- [ ] Change is based on evidence (book, case, data)
- [ ] CORE/AI balance is maintained
- [ ] Framework has template/checklist
- [ ] Example is provided
- [ ] Other agents notified
- [ ] Documentation updated
